## 1. Part 1 findings — detailed adjudication

### F-1. The lightweight read path — CONFIRMED, AND WORSE THAN STATED

Every element of the review's characterization checks out against the text, and the independent recount makes it *stronger*:

1. **The command count is three, not two.** The review counts `LocalAttemptRegistrationSpec::Autocommit` + `LocalReadCloseSpec`. It misses `LocalTerminalCompletionSpec`: Appendix A (line ~1457) states read-close "semantic success waits for the later `LocalTerminalCompletionSpec`," and §5.2 line 393 requires the outcome to reach `TerminalReady` "before any terminal result is returned." Three sequenced semantic commands per autocommit point read.
2. **Each command carries its own non-shareable order-attempt supplement.** The App A `ControlCommand` schema (line ~1835) mandatorily includes `order_attempt_input_ref` + availability-certificate ref for *every* semantic command, not just `CommitCommand`; each `LocalOrderAttemptInput` pins `expected_next_logical_command_seq`, so supplements structurally cannot be shared or batched across commands. The certificate plan/lock/share/publication steps consume *additional* maintenance-log positions (lines ~1485, ~1681). Total Raft log traffic per point read: **~6–10 entries**, not 2.
3. **The failure mode is worse than "saturates 8–16 cores."** Semantic apply is one strictly contiguous sequential chain (§12.8 line 927: `pipeline_order` strictly contiguous; entry *i* byte-matches candidate *i−1*). It cannot be spread across cores *at all*. At a realistic 1–10 µs per apply (BLAKE3 digest verification, canonical decode, multiple authenticated persistent-map CASes, pipeline-entry construction with `RaftGapClassificationProof`), the ceiling is ~33k–333k point reads/s — a **24×–240× shortfall** against the 8M/s gate. The unloaded protocol latency floor is ~50–150 µs (2–3 sequential fsync-backed order barriers) against a 15 µs p99 gate; under gate-level load, queueing on the serial chain pushes p99 to hundreds of µs.
4. **No carve-out exists.** Verified exhaustively: embedded returns only audit-visible `PublishedResultStream` (§13 line 959); `:memory:` runs the same state machine (line 381); explicit read transactions still register+publish per statement (App A line ~1411); only lifecycle-scoped admissions can create `AuthorizedSnapshot` cursors (App A line ~2548); all four "fast path" mentions in the plan are MVCC/SIMD internals, not lifecycle carve-outs. §17 law 1 ("no benchmark-only semantics," line 1205) closes the microbenchmark escape hatch. The one textual ambiguity — reading "8M lookups/s" as intra-statement probe throughput — was attacked and fails (it makes the attached p99 vacuous and violates the manifest discipline at line 1179); but the ambiguity itself should be named and closed by F-1-c.

**The internal contradiction is real: the gates and the protocol cannot both be true.** Severity "highest" is correct.

**Revision F-1-a (SnapshotQuery)** — adopt; it is the *only* viable shape. A delivery-policy toggle cannot help because the dominant cost is the three sequenced commands, not the retention records; any class keeping even one sequenced command per read is capped at the sequencer's apply rate. Required modifications the review omits:

1. **Fail-closed under Required-read audit profiles** (§12.8 lines 919/953 force this): reject with a typed, data-independent error steering to the durable class; never silently upgrade. A "ticketed-but-ephemeral" arm is coherent but pointless for v1 (audit-before-output dominates its latency anyway) — record it as a possible later arm.
2. **Security-barrier compliance**: during `AppliedHidden`, SnapshotQuery admission returns the same bounded backpressure class as everything else (line 929 forbids *all* new query execution, not just applied-tail-dependent work).
3. **Full registry citizenship**: new `ReadAccessAdmissionRecord` execution-scope arm, `command_type_classification` rows, App D message family, `AnswerContract` behavior row (non-resumable, no replay), FG-INV-04/20 clause amendments plus oracles.
4. **An explicit serializability argument with its own oracle** — in fact the class is *strict*-serializable by construction (visible-endpoint pin + ack-after-visibility, lines 389/560); state it, don't inherit it. ("No correctness is inherited merely by analogy" is the plan's own rule.)
5. **Bind-time class selection** from `CanonicalStatementClassification` (ProvenReadOnly, single-statement, within ephemeral output bounds) + delivery contract + audit profile — reusing existing machinery rather than a new API verb — and **excluding DP/Warden-classified statements** (the classification already carries `required_warden_operation_classes`).
6. **Policy/namespace-driven eligibility**, so governed namespaces can force all reads onto the durable audited path; plus sampled Sextant decision-card evidence on the fast path so the calibration plane doesn't go blind on the hottest surface.

**Revision F-1-b (fused autocommit write)** — adopt with two corrections. The review's "reservation + registration + publication + completion in one arm" is impossible: `TerminalCompletion` consumes durability/visibility evidence of the commit's *own* publication, and FG-INV-07 forbids a pre-order input naming its future applied record. The honest floor is **two** sequenced commands (fused Register+PrepareAdmit+Reserve+Commit, then TerminalCompletion) — a ~2.5× sequencer relief, not ~5×; **three** under a Required-audit profile (ticket admission must be audit-visible before protected processing, line 919). Second correction from the adversarial pass: the fused arm violates App A's "no statement/storage observation may precede visible registration" rule (line ~1409), since the statement must execute before the fused command orders. Adopting F-1-b therefore requires *amending that rule for the fused class* (coordinator-local admission satisfies FG-INV-20; durable registration lands atomically with commit; pre-order crash = re-execute from scratch, safe because nothing escaped and the idempotency-key CAS at apply preserves at-most-once effect). That is a lifecycle-rule change, not mere packaging — say so in the revision.

**Revision F-1-c** — trivially correct and required regardless. Fold into F-2-a, and add two things the review implies but never names: a standing §17 metric of **"sequenced semantic commands and total Raft log entries per operation, by operation class"** (the exact quantity whose absence let the contradiction survive), and a per-gate declaration of the **counting unit** (client-visible requests vs internal probes).

### F-2. Durability-barrier anatomy — CONFIRMED, UNDERCOUNTED

The five-barrier enumeration is verified and is a *lower bound*. At the plan's own mechanism granularity the smallest audit-Required write chain is **7–10 ordered sync points**: the review misses (a) the universal certificate protocol's durable steps — `CertificateAttemptPlan` quorum-rooting and per-signer `CertificateSignerLock` installs for *both* the payload and order-attempt availability certificates (App A line ~1495); (b) barriers 4 and 5 each being two ordered publications (quorum-durable maintenance entry, then protocol-root slot update, §12.8 line 945); and — from the adversarial pass — (c) for Required work, the **audit-ticket admission chain** (external CAS ticket issuance + ordered local `AuditTicketAdmissionSpec` + admission audit-visibility, line 919: a *second* external RTT before the counted barriers even start) and the audit event/resolution each being built under the plan/locks/shares signing protocol. All barriers terminate in the single serialized per-inode `RootPublicationSequencer` (line 334). The ~100 µs NVMe floor is conservative; cloud-block estimates of ~0.5 ms are optimistic (several ms is realistic).

Three corrections to the review:

- **Raft hard-state cost is per append *batch*, not per entry** — group commit batches entries into one hard-state publication. The framing is exact only at concurrency 1 (which is, however, the embedded wedge's hot case). The proposed gate should read "ordered sync operations per commit and per append batch, by posture and audit class."
- **"Quietly exclude" is unfair**: §12.8 line 947 explicitly mandates a separate governance-profile benchmark reporting visible-prefix lag and head-of-line saturation. What's missing is *gate status* (bench binary, committed baseline, variance budget), not acknowledgment. F-2-c is a promotion, not a discovery.
- **Barrier-fusion legality is unstated, and that is the sharpest argument for F-2-a**: in multi-member mode append-publication and D2 apply-publication genuinely cannot merge (followers must reply pre-commit); in quorum-one, a single fused publication arguably already satisfies every stated requirement — the text is silently permissive. This unpriced, unstated fusion opportunity is likely the biggest cheap latency win for the embedded posture.

**Revisions.** F-2-a: adopt at P0, but **not as a hand-maintained prose appendix** — the plan's own constitution (line 284: "hand-written lists… are never a second source of truth") demands a machine-checkable cost-model registry, generated where possible from `command_contracts.toml` publication modes and state slots, with µs budgets in `slo.toml`-class rows and a CI check that every §17 latency gate ≥ (its class's barrier count × pinned measured fsync p50). It must also settle barrier-fusion legality normatively (including the quorum-one append+apply fusion). F-2-b: P1 — adopt the sync-count gate unconditionally; evaluate the quorum-one fusion *first* (no new mutable state, no recovery-doctrine change — the crash window it creates is already handled by line 381's abandonment arms); run the `RaftHardFrame` spike only if multi-member follower append latency still justifies it, with the adversarial pass's correction in scope: three of `RaftHardState`'s five StrongRefs (certificate-attempt ledger, order-attempt root, audit signer journal) churn per commit, so a scalars-only mutable frame buys almost nothing unless those roots' cadence is restructured too, and skipping per-batch root publication requires a log-tail bootstrap (recovery-relevant bytes outside the root closure — new FG-INV-08/10 clauses). F-2-c: P1, decomposed at the plan's own two-plane boundaries (order-accepted→released, released→ActiveDeliverable→first-byte), run against a **pinned simulated audit authority under fgdb-sim** (satisfying line 1179's no-moving-comparators rule), derived from F-2-a.

### F-3. The unsized control/consensus stream — CONFIRMED, CONSERVATIVE

Verified at the format level, with refinements in both directions:

- **Per delivered subscription delta: ~3 dedicated same-group Raft maintenance entries** (publish / activation / ACK) in Required postures, ~2 in NotRequired. The review's 30–40k/s overstates slightly because **audit release is already batched** inside `AuditVisibilityAdvanceSpec` (its `sorted_protected_output_release_recipes[]` covers all pending outputs in a pipeline range) — the corrected figure at 10k cursors × 1 delta/s is ~20–30k entries/s. Same order of magnitude; conclusion unchanged.
- **The review's scenario is conservative in two ways it missed**: fan-out amplification (one commit matching C cursors mints ~3C entries — load scales with delivery cadence, not commit rate), and the §17 subscription gate itself (p99 < 10 ms) bounding the legal coalescing window, implying up to ~100 deliveries/s per hot cursor (upper bound; the per-cursor publish→release→activate→ACK serialization caps real cadence below that).
- **The `Coalesced` delivery-provenance arm (§9.5 line 753) does not rescue the arithmetic**: it reduces commits-per-delivery (fan-in), not transitions-per-delivery. Every coalesced envelope still pays the full publish/activate/ACK cycle.
- **The binding cost is the time authority, not the log.** Each delta activation requires a fresh `SubscriptionDeliveryLeaseTimeBasis` whose `TimeValidationEvidence::Usable` demands a challenge-nonce-fresh `PortableTimeAuthorityObservation` threshold-signed over *that exact subject digest* (App A lines ~2084–2090). At scenario scale that is ~10k challenged threshold-signed external observations per second — beyond any plausible threshold-signing service, and with no local-clock carve-out anywhere in the plan. The review's "the guard-budget machinery already exists" overstates: `maximum_guard_use_ticks` amortizes only within one subject and one process incarnation.
- **The missing cursor-scale gate is not merely advisable — it is doctrinally mandatory**: line 1179 requires every gate manifest to pin "transaction size and concurrency," so the line-1195 subscription gate without a cursor-count quantifier violates the plan's own gate-activation rule.

**Revisions.** F-3-a at P0: a normative capacity model (entries/s by class as functions of stated scale parameters, in the FG-CFG registry class — FG-CFG-03 is the in-plan template) plus a quantified §17 cursor-scale gate, stating the fan-out amplification factor and implied time-observation rate. F-3-b at P1, **reshaped**: not class-interval or window-derivation observations (which dilute the subject-bound property), but a registered **batch-subject arm** — `TimeBoundSubject::ActivationBatch` whose transcript is the sorted digest set of member lease-basis transcripts, one threshold-signed observation per activation tick, per-member window checks done by the batch checker (direct in-plan precedent: `TicketsResolvedBatch`'s one-entry-range discipline, and `OfflineMacaroonIssuerEpochCohort`'s set-shaped subject with per-member derivation proof). The adversarial pass's condition: land it **fused with F-3-c** (the batched activation spec consumes one evidence whose subject *is* the batch) rather than as a standalone evidence-derivation rule change. F-3-c at P1: `ProtectedOutputActivationBatchSpec` / `ProtectedOutputDispositionBatchSpec` with sorted per-subject sub-records, per-sub-record receipts/evidence, all-or-nothing apply with coordinator pre-filtering — this breaks neither the one-current-record/one-outstanding-lease laws nor FG-INV-15's per-record CAS discipline, and `AuditVisibilityAdvanceSpec`'s sorted release recipes are the in-plan precedent for one maintenance entry transitioning N protocol records. Single-item operations become canonical one-entry batches (no dual encoding). Go/no-go for the new rows belongs in the same G0 batch as F-3-a (maintenance rows are durable-format surface).

### F-4. W12 batch certification — CONFIRMED DIRECTION, UNDER-SPECIFIED LAW

The quotation and the serial reading are exact; the throughput framing is right and slightly *understated* (the true serial chain is certify → decide/build audit gate → order → apply, and — a point surfaced only in the adversarial pass — the plan never defines "fully applied" relative to Raft durability, so under the standard apply-after-commit reading, consecutive Meta semantic commands can never even share an fsync batch, making §14.6's own group-commit allowance vacuous for them). Two label corrections: "Calvin-style" is imprecise (Calvin has no certification phase; the correct precedents are FoundationDB's resolvers, Hyder's meld, Silo epochs — the substance "standard practice" stands); and the §20 risk-row "tension" is softer than my first pass suggested (the row says "the protocol is fixed in §14.6," so its "batch/pipeline" mitigation reads as the lawful I/O batching) — the strongest argument is simply that a normative law serializes certify→apply globally while no gate prices the ceiling.

**The review's law (complete intra-batch edges + canonical apply order) is necessary but not sufficient.** The plan's certification predicate is strictly larger than acyclicity. A sound batch law must additionally cover: FCW/uniqueness (an acyclic intra-batch ww pair still requires the canonical-later writer to abort — the deterministic loser priority at §14.6 already supplies the tie-break); constraint aggregates/cardinality (two members individually valid against the prefix can jointly violate a cap — needs an ordered fold, not pairwise edges); expected-branch-head drift (intra-batch same-head members: deterministically abort the canonical-later member via the existing drift rule — the "virtual head chaining" alternative is unsound unless `WeakMarkerIdentity` is proven sequence-independent, since markers are built only at apply); FG-INV-02's exclusive final-certification-basis ownership (clarify or re-scope the row to batch-scoped ownership with per-candidate sub-holds); and frontier recovery (a failed predecessor batch voids dependents through **admitted abort + new registration generation**, per the existing drift rule — not same-reservation recertification). Favorably: escrow (exclusive Held custody pre-certification), sequence/HLC assignment (counters live only in `MetaOrderAttemptInput`), and audit-event bases (supplements aren't signed into events) are already batch-safe by design.

For true pipelining (certify batch n+1 while batch n applies), the law must permit certification against the applied prefix **plus the decided-but-unapplied frontier** with complete cross-batch edges — sound because outcomes are final at certification and apply is mechanical ("The Meta record alone decides outcome"), but a stronger law than the review wrote down. Without it the relaxation only amortizes certification. Residual ceiling after the relaxation: ~1/apply (Meta apply's per-command sequence/HLC/marker/root digest chain stays serial) — publish both ceilings, as the plan's risk row already demands.

**Adopt as a "virtual-serial batch certification" law at P1, decided now** (W12 laws are normative now, formats reserved now; writing it at plan time is strictly cheaper than a post-1.0 format-epoch migration). The unified text must also record explicitly that this is *not* the enhancement-doc's rejected "Calvin deterministic transaction lane" — it is a batching law inside the existing certifier — so a future review doesn't relitigate that rejection.

### F-5-a. DirectoryBound allocation fence — CONFIRMED CONTRADICTION, REMEDY NEEDS REWORDING

The hole is real and internal: §4.5 unconditionally requires an external threshold-signed `IdentityContinuityRecord` before *any* epoch mints an ID — including the first VId of a fresh embedded database — while the plan already registers a **local-only DirectoryBound continuity arm for the cluster-incarnation record** (App A: "DirectoryBound uses no external record ref and a fixed local continuity digest") and forbids same-identity restore in that posture outright. No carve-out exists on any of the 97 `DirectoryBound` mentions, and FG-INV-11 hardcodes "externally monotonic." Read literally, `fgdb::Database::open(path)` requires an operator HSM or witness set before the first vertex — flatly contradicting the §1.5 "usable like DuckDB" promise. This is a genuinely new finding class: a seam *between* two absorbed hardening moves from the earlier Sol audit.

**But the review's proposed derivation is a false safety claim as worded.** Inode continuity (device, inode, filesystem generation) defeats *copy-based* restore, not **block-level rollback** — an LVM/ZFS/VM snapshot revert or dd-back preserves inode identity exactly, which is precisely why §4.5 put the record "outside the rollbackable database closure." Two facts from the adversarial pass sharpen the resolution: the external fence *itself* cannot detect intra-epoch rollback in any posture (the record binds only epoch + predecessor digest, no high-water marks), and its protective value in DirectoryBound is confined to a restore path that posture already forbids — so the local carve-out costs almost nothing in real security *within DirectoryBound's declared threat model*, but it must never be labeled "anti-rollback."

**Correct revision:** register an `AllocationContinuityProfile` bound 1:1 to the `IncarnationContinuityProfile` (ExternalCas ⇒ exactly today's rule; DirectoryBound ⇒ a local allocation-continuity journal under the same whole-inode fence, mirroring the existing fixed-local-digest pattern; W12/Meta require ExternalCas); scope FG-INV-11's "externally monotonic" per profile with a **registered residual-risk row** (block-level rollback is outside DirectoryBound's declared threat model, allocation included — the same conditional structure FG-INV-10 already uses for filesystem profiles); reconcile `AllocationEpochReservation`'s unconditional external-floor fields for the DirectoryBound arm; optionally register a TPM-NV-counter intermediate posture; and publish the per-posture external-authority footprint table (shared deliverable with F-5-b). P0 — it changes registered formats and unblocks the embedded story.

### F-5-b. Authority lattice — ACCEPT THE PRICING, REJECT THE MECHANISM

The head-of-line quotes are exact, the ~ten-authority census is fair (arguably one short), the §20 risk table verifiably lacks an authority-availability row, and two aggravating facts the review missed are confirmed: **Required-ticket issuance is a strictly serial direct-CAS-successor protocol at the external authority** (each ticket advances the continuity head by one — the authority is a synchronous *sequencer* for Required-op admission, sitting on the path three times: issuance CAS, admission visibility, resolution batch share); and the **dead-authority escape hatch requires the same authority** (`CompletenessChanged` is CAS-installed by it) — a permanently unavailable audit authority has *no specified recovery path*. Add: the time authority also gates `IdRangeLease` use (allocation-bearing writes), window-amortized.

But "write availability = product of every critical-path authority's availability" conflates three different availabilities, and the corrected formulation should be the one documented: **apply-durability availability** (local + Raft only; applies continue behind a stalled Required entry up to pipeline limits), **visibility availability** (× audit authority, for Required-profile databases), **delivery availability** (× time authority per activation — which the plan deliberately keeps *off* the audit prefix). Several listed authorities (archive, reservation, catalog, restore journal, witnesses, identity continuity) are episodic. Numbers belong in `slo.toml` per the claim-type constitution.

**Reject per-tenant/per-domain audit pipelines as specified.** The claim "the pipeline is keyed by role_and_group; adding a domain axis is an additive registry change" is architecturally false: pipeline entries are **cumulative role state roots in one linear chain** (entry *i* byte-matches candidate *i−1*), so releasing tenant B's entry k+1 past tenant A's unresolved Required entry k would expose a state root *containing* A's unaudited effect — no-leapfrog is forced by cumulative roots, not policy. The security barrier compounds it: authorization state is database-global, and line 929's noninterference argument depends on one total admission order. Independent per-domain visibility requires per-domain candidate roots — i.e., partitioning canonical state, a W12-scale redesign. The honest isolation unit the plan already provides is the **database** (per-lineage `AuditDomainId`; multi-database `fgdbd`) — the plan should say so normatively, mitigate within one database via per-tenant resolution SLOs at the authority plus the existing reserved capacity, and either register intra-database audit domains as a rejection or schedule them post-W12. Accept the third sub-revision (authority HA/durability requirements) and add the missing **successor-authority migration path** for a permanently dead registry (threshold-authorized transition recorded as a permanent continuity event). P1.

### F-6. EId width — CONFIRMED DIAGNOSIS, WRONG REMEDY, BETTER ONE AVAILABLE

The 16 GB arithmetic is exact ((16−8) B × 1e9 edges × 2 directions), the decision genuinely lacks a cost model, and the finding is *stronger* than stated: at raw widths a Tier D entry is ~42–46 B, so the plan's own "about 4 KiB and holds at most 256 entries" block (§6.2) reaches its 256-entry cap only at ≤16 B/entry — **the plan's own numbers already silently presume compressed identity encoding**, and "canonical stable-key encoding" occurs exactly once and is defined nowhere.

**The 64-bit remedy fails arithmetic the review never did.** Partition (20) + slot (44) already fill 64 bits; any 64-bit EId must shrink epoch, partition, or slot. e=16/s=28 gives 268M slots per (partition, epoch): at the §17 bulk-load gate a hot partition exhausts in ~7 seconds, forcing externally-CAS-fenced epoch advances onto the ingest path — precisely the external-authority coupling the review's own F-5 attacks — compounded by never-recycled abandoned leases. e=8/s=36 leaves ~256 epochs of terminal headroom against a rule where every write-enabled same-identity restore burns an epoch. Dropping the partition field instead means redesigning the allocator/lease/escape object model (`IdRangeLease` and `AllocationValueKind::EdgeId{graph,partition}` hardwire it). No 64-bit split works without more format surface than the problem is worth.

**The optimal fix keeps the 128-bit logical identity and prices the physical encoding**: normatively define "canonical stable-key encoding" for identity columns in §6.2/§6.3/App A as an order-preserving shared-prefix layout — block/run-header dictionary for `allocation_epoch`+`partition`, fixed-width or delta/FOR-coded slot suffix — registered through the existing `codec_table`, covered (as §6.3 already provides) by the logical-digest-over-stable-IDs validator so codecs evolve without format-identity change. Within partition-scoped blocks the prefix is near-constant; ~6 B/EId in Tier D and ~2–5 B in sorted sealed runs is realistic (reverse-family blocks are the caveat: destination-grouped entries carry source-partition EIds, degrading dictionaries under adversarial prefix diversity — typical-case savings survive). This saves *more* than the 64-bit option (10–14 B vs 8 B per entry), at zero semantic cost, and simultaneously repairs the latent 4 KiB/256 inconsistency. Add "identity-column bytes per live edge (by tier)" to §6.6/§17 law-4 reporting, and record the 128-bit width as final *with its real rationale* (restore epoch fencing, W12 partition mobility, shared branch allocator, never-recycle headroom) rather than symmetry. **Treat as G0-deadline work** — the definition lives in the format appendix that freezes there.

### F-7. Byte economy — CONFIRMED, WITH ONE MATERIAL CORRECTION TO THE ENDORSED FIX

## 3. Part 4 notes — four adopted, one refuted

**N-1 (strict serializability) — the review's one materially wrong technical finding.** Its preconditions *do* hold by construction (snapshot acquisition reads the current slot and revalidates under the lease; acks follow gap-free visible publication; applied-hidden commits are never acked), but the inference fails: the plan's certification law is **exact DSG acyclicity** (FG-INV-05; "rejects… any cycle"), with no real-time edges and no Cahill-pivot conservatism.
